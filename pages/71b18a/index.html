SELinux: Security-Enhanced Linux

安全上下文格式

Security Context 安全上下文 格式：user:role:type:sensitivity[:categories]

在Android中，通常可以忽略user、role、sensitivity这几个字段，因为Android并没有启用SELinux提供的所有功能

Android中的Security Context：

  1. user只定义了一个u,必要时，系统会通过categories字段来表示实际用户
  2. role只定义了2个：r来表示subjects(主体，如某个进程、服务)，object_r来表示objects（客体，如某个文件）
  3. 没有使用sensitivity，始终都是默认值s0
  4. categories配合sensitivity会出现多级安全权限问题MLS,

关键文件

Policy file

*.te文件时SELinux策略源文件，定义了domain和标签

Context files

上下文文件是为对象（客体）指定标签的文件，主要有以下几个

文件名 一般用途
file_contexts 为文件分配安全上下文
service_contexts 用于为 Android Binder 服务分配标签，以便控制哪些进程可以为相应服务添加（注册）和查找（查询）Binder 引用。在启动期间，servicemanager 进程会读取此配置
seapp_contexts 用于为应用进程和 其对应/data/data 目录分配标签。在每次应用启动时，zygote 进程都会读取此配置；在启动期间，installd 会读取此配置
property_contexts 用于为 Android 系统属性分配标签，以便控制哪些进程可以设置这些属性。在启动期间，init 进程会读取此配置
genfs_contexts 用于为不支持扩展属性的文件系统（例如，proc 或 vfat）分配标签
mac_permissions.xml 用于根据应用签名和应用软件包名称（后者可选）为应用分配 seinfo 标记。随后，分配的 seinfo 标记可在 seapp_contexts 文件中用作密钥，以便为带有该 seinfo 标记的所有应用分配特定标签。在启动期间，system_server 会读取此配置

SELinux policy

Android8.0 之后，SELinux policy分成了platform和vendor 两个部分，允许独立更新

platform sepolicy被进一步拆成了platform private和 platform public两个部分，来 暴露指定的类型和属性给vendor策略编写者。所以只有定义在public中的type和attribute才能在vendor中直接使用

Platform public sepolicy

平台公共策略文件，路径 system/sepolicy/public，这部分是用来导出对vendor可见的

其中PLATFORM_SEPOLICY_VERSION定义了vendor可见的版本，这个宏定义可以在build/make/core/config.mk中找到

Platform private sepolicy

平台私有策略文件，路径 /system/sepolicy/private,这部分vendor不可见

Android11开始又增加了system_ext policy 和 product policy，system_ext和product策略也区分公共策略和私有策略。

原生AOSP有以下几个目录：

路径 描述
system/sepolicy/public 平台公共策略一般是定义的一些type和attribute，对外可见
system/sepolicy/private 平台实现的具体策略，外部不可见
system/sepolicy/vendor 供应商可以使用的政策和上下文文件

另外还有一些定义的宏，编译时，会去找这些宏定义的目录下的策略文件：

变量 含义
BOARD_SEPOLICY_DIRS 供应商的 sepolicy目录，但是在sepolicy的Android.mk中看到这个已经被废弃了，实际是用的BOARD_VENDOR_SEPOLICY_DIRS
BOARD_ODM_SEPOLICY_DIRS Odm sepolicy
SYSTEM_EXT_PUBLIC_SEPOLICY_DIR System_ext公共策略，对vendor可见 （Android11之前用的是BOARD_PLAT_PUBLIC_SEPOLICY_DIR）
SYSTEM_EXT_PRIVATE_SEPOLICY_DIRS System_ext 实现的具体策略，对外不可见（Android11之前用的BOARD_PLAT_PRIVATE_SEPOLICY_DIR）
PRODUCT_PUBLIC_SEPOLICY_DIRS Product公共策略，对vendor可见
PRODUCT_PRIVATE_SEPOLICY_DIRS Product实现的具体策略，外部不可见

构建系统会采用以上定义的policy，在对应的分区system、system_ext、product、vendor、odm下生成对应的策略组件

  1. 将策略转换为 SELinux 通用中间语言(Common Intermediate Language) (CIL) 格式，具体如下

    a. 公共平台策略（包含system、system_ext、product）

    b. 私有 + 公共组合

    c. 公共+ vendor and BOARD_SEPOLICY_DIRS policy

type、attribute访问权限

本地验证发现：

  1. 平台的公共部分，system_ext、vendor、product中都可以引用到
  2. 在product定义的public部分，system_ext访问不到，vendor可以访问。（如 在 product中定义的carservice_app，system_ext访问不到，但是vendor可以访问）
  3. 在system_ext定义的public部分，product和vendor都可以访问
  4. system中的只能访问system中的定义的

在system中新增type和attribute的兼容性处理

修改system/sepolicy/public或system/sepolicy/private下的策略时，会出现兼容性问题，编译会报 sepolicy_freeze_test，因为selinux权限有校验，修改后的文件会和预编译目录/system/sepolicy/prebuilts/api/30.0/下的文件进行比较，文件内容不同，校验不通过，所以需要将/system/sepolicy/private和/system/sepolicy/public修改过的地方，同步在/system/sepolicy/prebuilts/api/30.0/对应文件中进行修改。

另外如果在system/sepolicy/public或system/sepolicy/private中新增了type或attribute，还需要对新增的进行忽略，编译才能通过：

在/system/sepolicy/private/compat 目录下，找到最新的api目录，修改其中的ignore.cil，将新增的type加入到new_objects，不做兼容性校验。记得同步更新prebuilts/api/30.0/private/compat下对应文件。

Android中定义好的attribute

Android中定义的attribute可以在system/sepolicy/public/attributes中查看.

常用的如下：

attribute 含义
dev_type 所有设备类型
domain 所有进程类型
appdomain 所有app进程
file_type 所有文件类型
exec_type 所有可执行进程入口
data_file_type 所有/data 目录下的文件
system_file_type 所有/system目录下的文件
vendor_file_type 所有/vendor目录下的文件
service_manager_type 所有通过servicemanager管理的服务
hwservice_manager_type 所有通过hwservicemanager管理的服务
mlstrustedsubject 所有能越过MLS检查的主体（进程）
mlstrustedobject 所有能越过MLS检查的客体（低权限用户可读，高权限用户可写的文件）

selinux权限配置

mlstrustedsubject

selinux权限的宏定义：global_macros

快速配置权限可以使用audit2allow：https://source.android.google.cn/security/selinux/validate?hl=zh_cn#using_audit2allow

添加权限容易，但实际中会遇到和Android selinux权限策略冲突，编译无法通过（neverallow问题）。

neverallow的问题解决思路是，在对应的地方添加 过滤。将冲突的地方添加例外。

但由于兼容性检查，需要处理当前api版本和前一个api版本的selinux权限

以adasService为例：

  1. 运行时报错， init 没有 通过process对 adasService进行 transition的权限

  2. 直接allow init adasService:process transition;是不行的，编译无法通过，违法了neverallow规则。

  3. 找到对应的neverallow规则

    添加例外，将adasService添加进去

  4. 此时规则正确，但selinux校验会失败，编译会报 sepolicy_freeze_test ，因为selinux权限有校验，修改后的文件会和/system/sepolicy/prebuilts/api/30.0/中进行比较，文件内容不同，校验不通过，所以需要将/system/sepolicy/private和/system/sepolicy/public修改过的地方，同步在/system/sepolicy/prebuilts/api/30.0/对应文件中进行修改

  5. 还有兼容性问题，在/system/sepolicy/private/compat 目录下，找到最新的api目录，修改其中的ignore.cil，将新增的type加入到new_objects，不做兼容性校验。记得同步更新prebuilts/api/30.0/private/compat下对应文件。

hal_vehicle neverallow

libsepol.report_failure: neverallow on line 6 of system/sepolicy/public/hal_vehicle.te (or line 20021 of policy.conf) violated by allow avm3d_service hal_vehicle_hwservice:hwservice_manager { find };

居然出现了neverallow，hal_vehicle是android原生的车载服务标签，查看对应的te，发现没有限制。跑去对比其他te发现是需要添加：

hal_client_domain(avm3d_service, hal_vehicle)

即需要跟hal_vehicle进行client绑定
